---

## 版本号规范

### 语义化版本 (Semantic Versioning)

项目采用语义化版本号格式：`MAJOR.MINOR.PATCH`

- **MAJOR（主版本号）**：不兼容的 API 变更
- **MINOR（次版本号）**：向下兼容的功能性新增
- **PATCH（修订号）**：向下兼容的问题修正

### 版本号示例

```
1.0.0          # 初始版本
1.0.1          # 修复 bug
1.1.0          # 新增功能
2.0.0          # 重大变更，不兼容
```

### 预发布版本

预发布版本使用以下格式：

```
1.0.0-alpha    # Alpha 版本
1.0.0-beta     # Beta 版本
1.0.0-rc.1     # 发布候选版本 1
1.0.0-rc.2     # 发布候选版本 2
```

## 发布流程

### 1. 发布准备

1. **更新版本号**
   ```bash
   # 修改 manifest.json 中的版本号
   "version": "1.1.0"
   ```

2. **更新 CHANGELOG**
   - 记录本次发布的所有变更
   - 按功能分类整理变更内容
   - 标注重要变更和破坏性变更

3. **运行测试**
   - 确保所有测试通过
   - 进行集成测试
   - 验证关键功能

### 2. 打包发布

1. **清理构建**
   - 移除临时文件
   - 清理调试代码
   - 确认生产环境配置

2. **打包扩展**
   ```bash
   # 创建发布包
   zip -r YiPet-1.1.0.zip . -x "*.git*" -x "docs/*" -x ".DS_Store"
   ```

3. **验证包内容**
   - 确认包文件完整
   - 测试加载扩展
   - 验证功能正常

### 3. 发布执行

1. **创建 Git 标签**
   ```bash
   git tag -a v1.1.0 -m "Release version 1.1.0"
   git push origin v1.1.0
   ```

2. **创建 GitHub Release**
   - 上传打包文件
   - 填写发布说明
   - 关联相关 Issue

3. **发布到 Chrome Web Store**
   - 上传扩展包
   - 更新扩展描述
   - 提交审核

### 4. 发布后

1. **通知相关人员**
   - 团队内部通知
   - 用户通知（如需要）

2. **监控运行状态**
   - 观察错误率
   - 收集用户反馈
   - 监控性能指标

## 回滚机制

### 紧急回滚流程

1. **停止发布**
   - 立即停止当前发布
   - 通知相关人员

2. **评估影响**
   - 确认问题范围
   - 评估回滚必要性

3. **执行回滚**
   - 使用上一个稳定版本
   - 重新发布
   - 验证回滚结果

4. **分析原因**
   - 找出问题根源
   - 制定改进措施
   - 更新发布流程

## CHANGELOG 规范

### CHANGELOG 格式

```markdown
## [1.1.0] - 2026-03-18

### 新增
- 新增宠物角色系统
- 添加 Mermaid 图表渲染功能

### 改进
- 优化聊天界面性能
- 改进截图区域选择

### 修复
- 修复宠物拖拽问题
- 修复 API 调用错误

### 破坏性变更
- 配置文件格式变更
```

### 变更类型

- **新增**：新功能或新特性
- **改进**：现有功能的改进或优化
- **修复**：bug 修复
- **破坏性变更**：不兼容的变更

## 最佳实践

1. **小版本发布**：频繁发布小版本，降低风险
2. **充分测试**：发布前进行充分测试
3. **详细文档**：记录每个版本的变更
4. **准备回滚**：始终准备好回滚方案
5. **及时沟通**：发布前与团队充分沟通
