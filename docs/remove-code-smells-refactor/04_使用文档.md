## 功能介绍

本项目对 YiPet 温柔陪伴助手进行代码味道清理和模块化重构，消除重复的 ErrorHandler 和 LoggerUtils 类、简化过度细分的 pet 模块、统一文件命名规范、将中文目录名改为英文名、清理 screenshot 死代码。重构后的代码库更易维护、更一致，同时保证 Chrome Extension（Manifest V3）的功能完整性。

- 🎯 **代码库更精简**：消除重复代码，移除死代码
- ⚡ **模块结构更清晰**：合理合并过度细分的文件
- 🔧 **命名规范统一**：所有文件使用 kebab-case 命名

## 常见问题

### 💡 基础问题

**Q1：为什么 ErrorHandler 和 LoggerUtils 只有一个实现了？**

重构前有两个重复实现，容易混淆使用哪个。重构后合并为一个，消除代码冗余，避免维护两套实现导致的 Bug 不一致。

**Q2：为什么 pet 模块文件数量减少了？**

之前过度细分到 24+ 个文件，职责边界不清晰。重构后合理合并到 <15 个文件，降低认知负担，提升可维护性。

**Q3：为什么所有文件名改成了 kebab-case？**

之前命名不统一，混用 petManager.*.js、export-chat-to-png.js、injectionService.js 等多种风格。重构后统一使用 kebab-case，提升代码一致性和可读性。

**Q4：为什么中文目录名改成了英文名？**

中文目录名（医生、教师、甜品师、警察）存在跨平台编码风险。重构后改为 doctor、teacher、chef、police，路径在所有环境下稳定可解析。

**Q5：为什么 screenshot 模块删除了？**

README.md v1.1.0 提到"移除截图功能"，但目录仍存在造成困惑。重构后确认废弃并清理，减少代码库负担。

### ⚙️ 高级问题

**Q6：如何确认 manifest.json 和 injectionService.js 完全一致？**

对比两个文件的脚本列表，必须逐行完全一致。不一致会导致 PetManager 未定义等运行时错误。

**Q7：添加新文件时如何确保命名规范？**

使用 kebab-case 命名文件，类名保持 PascalCase。参考现有文件的命名风格。

**Q8：如何验证所有路径引用已更新？**

搜索代码库中可能的引用，检查 Chrome DevTools Network 面板是否还有 404 错误。

### 🔧 故障排查

**Q9：重构后扩展加载失败怎么办？**

检查 manifest.json 语法是否正确，检查所有文件路径是否存在，检查 Chrome DevTools Console 错误信息。

**Q10：重构后宠物不显示怎么办？**

检查 petManager 加载顺序是否正确（core 在前，扩展在后），检查图片路径是否正确，检查 Network 面板是否有图片 404。

**Q11：角色切换后图片不显示怎么办？**

确认角色目录名使用英文名，确认 imageResourceManager.js 中路径前缀正确，确认 chrome.runtime.getURL() 参数正确。

## 技巧与提示

### 💡 实用技巧

1. **两个文件同步更新**：修改任何文件路径后，同步更新 manifest.json 和 injectionService.js
2. **使用 git mv**：重命名文件时使用 git mv 保留 git 历史
3. **英文命名**：所有目录名使用英文小写，避免中文
4. **分阶段验证**：每完成一个重构阶段就验证一次，避免问题堆积
5. **保留 Git 历史**：使用 git mv 而不是手动删除新建
6. **充分测试**：重构后充分测试所有核心功能

### ⌨️ 快捷键

### 📚 最佳实践

- 新增文件时遵循 kebab-case 命名规范
- 修改路径后立即验证 Chrome 扩展是否正常加载
- 使用 `git mv` 移动文件，保留 git 历史
- 重构分阶段提交，每步可独立回退
- 重构后充分测试所有核心功能
- 保持 manifest.json 和 injectionService.js 完全一致
