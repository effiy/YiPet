## §1 概述

YiPet Chrome Extension 架构全景分析，固化模块拓扑、数据流、安全边界、依赖变更影响与新人上手指南。覆盖 content-script + service-worker + popup 三层架构，让后续故事规划、影响分析和架构决策有唯一事实参照。

面向的用户角色：架构师、开发者、安全审计者、新加入开发者。

## §2 Story

### Story 1: 模块地图与拓扑

作为架构师，我想要掌握 YiPet 全量模块拓扑和依赖关系，以便评估架构复杂度和模块变更的波及范围。优先级 P0。范围边界：覆盖 core/、modules/pet/、modules/extension/、modules/faq/ 全部模块的静态依赖关系。依赖：manifest.json 的 content_scripts 加载顺序声明。

#### 功能点与约束

| 类别 | 描述 | 输入/约束 | 输出/校验 | 错误/说明 | 优先级 |
|------|------|----------|----------|----------|--------|
| 功能点 | 模块识别与分类 | 项目所有 JS 文件 | 按目录层级分类的模块列表（core/lib/pet/ext/faq） | 遗漏模块 → 拓扑图不完整 | P0 |
| 功能点 | 依赖关系图 | manifest.json 加载顺序 + 源码 import/IIFE 引用 | 有向依赖图（mermaid flowchart） | 循环依赖 → 加载失败 | P0 |
| 功能点 | 全局命名空间映射 | 所有 IIFE 模块的 root 导出 | 导出符号 × 消费者矩阵 | 符号冲突 → 运行时覆盖 | P0 |
| 功能点 | 入口验证 | manifest content_scripts 数组 | 全部 88 个文件路径存在且可加载 | 缺失文件 → 扩展加载失败 | P0 |

#### 成功标准

- 🎯 **模块可遍历** — 度量：从任一模块出发可追踪其所有依赖和被依赖 · 目标：依赖关系图连通可遍历
- ✅ **符号可验证** — 度量：每个全局导出符号在 DevTools Console 中 `typeof !== 'undefined'` · 目标：关键符号全部可访问
- 🔍 **加载顺序无循环** — 度量：拓扑排序检测 manifest 加载顺序 · 目标：无循环依赖

### Story 2: 数据流追踪

作为开发者，我想要追踪用户操作到 API 端点、存储读写、Service Worker 消息路由的完整数据流，以便定位问题根因和评估性能瓶颈。优先级 P0。范围边界：覆盖 AI 聊天流、会话 CRUD 流、FAQ 查询流、storage 读写路径、SW 消息路由。依赖：ApiManager、RequestClient、TokenManager、chrome.storage API。

#### 功能点与约束

| 类别 | 描述 | 输入/约束 | 输出/校验 | 错误/说明 | 优先级 |
|------|------|----------|----------|----------|--------|
| 功能点 | 聊天请求链路追踪 | 用户输入消息 → API 响应渲染 | 序列图（用户→ChatInput→PetManager→AI→ApiManager→Token→RequestClient→fetch→API） | 链路中断 → 消息无法发送 | P0 |
| 功能点 | 会话 CRUD 数据流 | 创建/查询/更新/删除会话 | 序列图（前端→SessionService→ApiManager→远程 API + chrome.storage.local 同步） | 存储不可用 → 降级到内存 | P0 |
| 功能点 | chrome.storage.local 读写路径 | 所有 storage key 及其读写者 | 写入者×存储 key×读取者 矩阵 | 配额超限 → cleanupOldData 清理 | P0 |
| 功能点 | Service Worker 消息路由 | 消息 action → handler 映射 | 6 条消息通道 + action 路由表 | 未知 action → 返回错误 | P1 |

#### 成功标准

- 🎯 **链路可追踪** — 度量：聊天请求从用户输入到 API 响应的每一步皆有文档记录 · 目标：关键数据流可追踪到端点
- ✅ **存储路径完整** — 度量：全部 chrome.storage.local key 的读写者均有记录 · 目标：无遗漏的存储 key

### Story 3: 信任边界与安全面

作为安全审计者，我想要识别 YiPet 的全部信任边界，覆盖认证、传输、输入、存储、权限五大安全面，以便发现安全缺口并制定缓解策略。优先级 P0。范围边界：Token 认证、fetch 传输、用户输入、chrome.storage 存储、`<all_urls>` 权限。依赖：TokenManager、ApiManager、RequestClient、StorageHelper、manifest.json 权限声明。

#### 功能点与约束

| 类别 | 描述 | 输入/约束 | 输出/校验 | 错误/说明 | 优先级 |
|------|------|----------|----------|----------|--------|
| 功能点 | Token 认证审计 | TokenManager 三级降级逻辑 | 降级链路图 + 安全评级矩阵 | Token 无过期机制（缺口 P2） | P0 |
| 功能点 | 传输安全审计 | fetch CORS + X-Token 头 + HTTPS | 传输层安全机制矩阵 + 风险评估 | POST 重试非幂等（缺口 P1） | P0 |
| 功能点 | 输入安全审计 | 聊天/Token/FAQ/标题/AI 设置 5 条输入路径 | 风险标注图（P0/P1/P2） | 聊天消息无 XSS 过滤（缺口 P0） | P0 |
| 功能点 | 存储安全审计 | chrome.storage.local 所有 key 的敏感性评级 | 存储安全矩阵 + 配额清理流程 | 无数据加密（缺口 P2） | P1 |
| 功能点 | 权限安全审计 | manifest 5 项权限 + `<all_urls>` | 权限用途审查矩阵 + 缓解措施 | `<all_urls>` 范围过大（缺口 P1） | P1 |

#### 成功标准

- 🔒 **安全面全量覆盖** — 度量：五面审查矩阵每面 ≥ 3 条检查项 · 目标：安全面全量覆盖无绕过
- ⚠️ **缺口标注优先级** — 度量：每个已知缺口关联至缓解策略和优先级 · 目标：所有缺口可追踪

### Story 4: 依赖关系与变更影响

作为维护者，我想要量化每个依赖（外部 CDN + 内部模块）失效时的爆炸半径，以便评估变更风险并制定测试策略。优先级 P0。范围边界：7 个外部 CDN 依赖 + 10 类内部模块变更触发源。依赖：manifest 加载顺序、IIFE 全局命名空间、PetManager 原型链。

#### 功能点与约束

| 类别 | 描述 | 输入/约束 | 输出/校验 | 错误/说明 | 优先级 |
|------|------|----------|----------|----------|--------|
| 功能点 | 外部依赖失效分析 | Vue/marked/turndown/Mermaid/md5/Tailwind 6 个关键依赖 | 失效影响矩阵（影响范围 + 爆炸半径 + 降级行为） | CDN 不可达 → 功能降级 | P0 |
| 功能点 | 内部模块变更影响 | manifest 顺序/PET_CONFIG/PetManager/ApiManager 等 10 类触发源 | 影响文件数量 + 推荐验证方式 | 接口变更 → 连锁影响 | P0 |
| 功能点 | 依赖可用性检测流程 | 扩展初始化时的检测链 | 检测流程图（PET_CONFIG → Vue → marked → turndown → storage → runtime.id） | 任一失败 → 对应降级 | P1 |

#### 成功标准

- ⚡ **变更影响可量化** — 度量：每种变更类型的受影响文件数可估算 · 目标：变更影响可量化可追溯
- 🛡️ **降级行为明确** — 度量：每个外部依赖失效有明确的降级行为描述 · 目标：降级路径已知不意外

### Story 5: 新人上手与开发指南

作为新加入开发者，我想要独立完成环境搭建、掌握调试方法、完成首个代码修改，以便快速融入项目开发。优先级 P1。范围边界：环境搭建、3 种调试入口、代码模式、常见任务操作指南。依赖：Chrome 扩展开发者模式、项目源码结构。

#### 功能点与约束

| 类别 | 描述 | 输入/约束 | 输出/校验 | 错误/说明 | 优先级 |
|------|------|----------|----------|----------|--------|
| 功能点 | 环境搭建指南 | 项目源码 | 3 步搭建流程（git clone → 加载扩展 → 验证） | 无构建步骤（纯 JS 项目） | P0 |
| 功能点 | 调试方法指南 | Chrome DevTools | Content Script / Service Worker / Popup 3 种调试入口 | 上下文失效 → 刷新页面 | P1 |
| 功能点 | 代码模式速查 | IIFE 模块 / Vue Options API / Prototype 扩展 / 配置端点注册 | 4 种代码模式 + 示例 | 模式不一致 → 维护困难 | P1 |
| 功能点 | 常见任务操作指南 | 新增组件 / 新增端点 / 新增子模块 | 3 种常见任务的步骤清单 | 遗漏步骤 → 功能不可用 | P1 |

#### 成功标准

- 🚀 **新人可独立首修** — 度量：新人按指南完成环境搭建 + 首修的时间 · 目标：环境搭建 3 步完成，首修在合理时间内完成
- 📖 **文档可操作** — 度量：常见任务指南的步骤能否被独立复现 · 目标：所有步骤可独立复现
