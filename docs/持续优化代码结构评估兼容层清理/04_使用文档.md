### 📋 场景一：清理纯兼容层（petManager.ai.js 和 petManager.mermaid.js）

**适用情况**：需要清理无实际作用的纯兼容层文件

**操作步骤**：

1. **备份 Manifest**
   ```bash
   cp manifest.json manifest.json.backup
   ```

2. **修改 Manifest**
   - 打开 manifest.json
   - 注释掉第 51 行：`"modules/pet/content/modules/petManager.ai.js",`

3. **测试功能**

## 常见问题

### 💡 基础问题

**Q: 为什么要清理兼容层？**
A: 兼容层在重构过渡期有用，但长期保留会增加维护成本，造成代码位置混淆。清理后代码结构更清晰。

**Q: 为什么不直接删除兼容层文件，而是先从 Manifest 移除？**
A: 这是安全策略。先从 Manifest 移除引用，保留文件本身。如发现问题，可以快速恢复。确认稳定后再彻底删除。

**Q: petManager.editor.js 的异常是怎么产生的？**
A: 可能是前两阶段重构时的遗留问题。文件头部改成了兼容层，但内容没有完全迁移，导致现在的异常状态。

### ⚙️ 高级问题

**Q: 如何判断一个模块是否需要拆分？**
A: 参考设计文档中的评估标准：1) 文件大小（>1000行优先考虑）；2) 职责清晰度；3) 变更频率；4) 依赖关系；5) 与现有架构风格一致性。

**Q: 清理兼容层后出现问题怎么办？**
A: 使用备份快速回滚。然后分析问题原因，调整清理策略。可能需要延长兼容层的保留时间。

**Q: 这个阶段完成后，还会有第四阶段重构吗？**
A: 视情况而定。如果后续发现新的问题或有新的功能需求导致代码结构变化，可能会有后续优化。但本阶段目标是完成前两阶段的收尾工作。

### 🔧 故障排查

**Q: 从 Manifest 移除兼容层后，功能异常怎么办？**
A:
1. 立即从备份恢复 manifest.json
2. 检查是否有其他地方依赖兼容层的加载
3. 可能需要延长兼容层的保留时间
4. 详细分析依赖关系后再尝试

**Q: 修复 petManager.editor.js 时，不确定保留哪个版本怎么办？**
A:
1. 优先保留 editor/ 目录下的新模块（core.js 和 ui.js）
2. 详细对比功能完整性
3. 将旧文件中的独有功能迁移到新模块
4. 充分测试后再清理

**Q: 清理后发现有功能缺失怎么办？**
A:
1. 立即回滚到清理前状态
2. 详细分析缺失功能的依赖关系
3. 调整清理策略
4. 可能需要保留部分兼容层，或迁移相关代码

## 技巧提示

### 💡 实用技巧

1. **分步执行，步步验证**
   - 不要一次性执行所有清理
   - 每完成一项修改就充分验证
   - 确认正常后再继续下一步

2. **使用 git 分支管理**
   ```bash
   git checkout -b phase3-cleanup
   # 完成一项工作就提交一次
   git add . && git commit -m "清理 xxx"
   ```

3. **保留完整的备份**
   - 清理前创建备份分支
   - Manifest 修改前先备份
   - 重要文件删除前先备份

4. **充分利用控制台日志**
   - 清理前观察正常的日志输出
   - 清理后对比日志变化
   - 注意任何新增的错误或警告

### 📚 最佳实践

1. **评估优先，行动在后**
   - 先完成所有评估工作
   - 再制定详细的执行计划
   - 最后才开始实际修改

2. **保持沟通**
   - 如果是团队项目，清理前通知团队
   - 说明清理范围和可能的影响
   - 约定回滚策略

3. **文档同步更新**
   - 清理完成后更新相关文档
   - 记录清理过程和遇到的问题
   - 为未来类似工作提供参考

4. **渐进式清理**
   - 先清理风险最低的（纯兼容层）
   - 再处理复杂的（异常状态）
   - 给每个阶段足够的观察期
