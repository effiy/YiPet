## 功能介绍

YiPet 项目的模块化目录重构通过三层机制（目录边界、module.md 依赖清单、CLAUDE.md 层级约束）为 AI coding 提供上下文收敛能力。每个模块形成独立的内容边界，AI 修改时只需加载目标模块及其直接依赖，而非整个项目的 ~19000 行代码。

🎯 **上下文收敛**：单模块任务上下文从全项目缩减至 1-3 个模块（≤ 3000 行）
⚡ **依赖透明**：module.md 显式声明模块间依赖，影响链可精确追踪
🔧 **渐进适用**：可逐步为各模块创建清单，不需要一次性完成所有模块

本功能面向使用 AI coding 工具（Claude Code / Cursor）开发 YiPet 项目的开发者。

## 快速开始

### 前置条件

- ✅ YiPet 项目已克隆到本地
- ✅ 已安装 AI coding 工具（Claude Code CLI 或 Cursor IDE）
- ✅ 了解 Chrome Extension 基本架构（background / content / popup）

### 30 秒上手

1. 打开项目根目录的 `CLAUDE.md`，查看"模块地图"章节
2. 找到你要修改的目标模块名和目录路径
3. 进入目标模块目录，阅读 `module.md` 了解依赖范围
4. 让 AI 只在目标模块范围内执行修改
5. 修改完成后检查 `module.md` 的反向依赖确认无遗漏 🎉

## 操作场景

### 📋 场景 1：修改聊天窗口 UI

**适用情况**：需要修改 ChatWindow 组件的样式、布局或交互逻辑

**操作步骤**：

| 步骤 | 操作 | 说明 |
|------|------|------|
| 1 | 读取 `CLAUDE.md` 模块地图 | 定位 `modules/pet/` 模块 |
| 2 | 读取 `modules/pet/module.md` | 础认上游依赖为 `core/config`、`core/utils`、`core/api` |
| 3 | 让 AI 加载 chat 相关文件 | 仅加载 `modules/pet/components/chat/` + `modules/pet/content/petManager.chatUi.js` |
| 4 | 执行修改 | 上下文 ~2000 行而非全项目 |
| 5 | 检查反向依赖 | module.md 中 chat 的反向依赖为 FAQ 模块 |

**预期结果**：修改完成，ChatWindow 功能正常，其他模块不受影响

**注意事项**：
- ✅ 确保只在 chat 相关文件范围内修改
- ❌ 不要加载 pet/session/faq 等无关模块的完整代码

### 🎨 场景 2：修改 API 请求层

**适用情况**：需要修改 ApiManager、SessionService 或 FaqService 的接口

**操作步骤**：

| 步骤 | 操作 | 说明 |
|------|------|------|
| 1 | 读取 `CLAUDE.md` 模块地图 | 定位 `core/api` 模块 |
| 2 | 读取 `core/api/module.md` | 础认反向依赖为 `modules/pet`、`modules/chat`、`modules/faq` |
| 3 | 让 AI 加载 core/api + 受影响模块接口 | 加载 API 层 + 各模块的 API 调用代码 |
| 4 | 执行修改并验证兼容性 | 检查所有反向依赖模块的接口调用是否兼容 |
| 5 | 更受影响模块的 module.md | 如接口变更影响依赖声明，更新清单 |

**预期结果**：API 层修改完成，所有依赖模块的调用兼容

**注意事项**：
- ✅ 必须检查 core/api 的所有反向依赖模块
- ❌ 不要只修改 API 层而不验证依赖模块兼容性

### 🔍 场景 3：新增功能模块

**适用情况**：需要添加新的功能模块（如通知模块）

**操作步骤**：

| 步骤 | 操作 | 说明 |
|------|------|------|
| 1 | 按 module.md 模板创建目录 | 创建 `modules/notification/` 目录和 module.md |
| 2 | 填写 module.md 依赖声明 | 声明上游依赖（core/utils、core/api） |
| 3 | 在 CLAUDE.md 注册新模块 | 在模块地图章节添加新模块条目 |
| 4 | 更新 manifest.json | 添加新模块的 content_scripts 入口 |
| 5 | 创建模块代码 | 按模块边界独立开发 |

**预期结果**：新模块自动获得上下文边界约束，AI 可按需加载

**注意事项**：
- ✅ module.md 必须声明所有实际依赖
- ❌ 不要在 module.md 中遗漏对 core 层的依赖声明

### 📊 场景 4：移除功能模块

**适用情况**：需要移除某个不再需要的模块（如 screenshot）

**操作步骤**：

| 步骤 | 操作 | 说明 |
|------|------|------|
| 1 | 读取目标模块 module.md | 础认反向依赖声明 |
| 2 | 全项目搜索模块引用 | 验证无硬编码引用遗漏 |
| 3 | 从 CLAUDE.md 移除模块条目 | 删除模块地图中的注册 |
| 4 | 更新 manifest.json | 移除相关 content_scripts 入口 |
| 5 | 删除模块目录 | 安全移除模块文件 |

**预期结果**：模块安全移除，无遗留依赖断裂

**注意事项**：
- ✅ 必须先确认反向依赖为空或已全部处理
- ❌ 不要在未确认反向依赖的情况下直接删除模块

## 常见问题

### 💡 基础问题

**Q：module.md 是什么？**

A：module.md 是每个模块目录下的元数据清单文件，声明该模块的职责、入口文件、上游依赖、反向依赖和导出接口。它帮助 AI 确定上下文加载范围，不参与运行时。

**Q：module.md 会影响 Chrome 扩展的运行吗？**

A：不会。module.md 是纯 Markdown 元数据文件，不会被 Chrome 扩展加载。它只用于 AI coding 的上下文控制。

**Q：CLAUDE.md 的模块地图章节在哪里？**

A：在项目根目录的 `CLAUDE.md` 文件中，新增了"模块地图"章节。AI coding 工具会自动读取此文件。

**Q：每个模块都需要 module.md 吗？**

A：P0 阶段需要为所有功能模块创建 module.md。P1 阶段为核心模块（pet、chat、session）创建独立的 CLAUDE.md。

### ⚙️ 高级问题

**Q：module.md 的依赖声明如何保持与代码一致？**

A：依赖声明基于代码中的实际引用（grep 搜索 `window.PetManager`、`PET_CONFIG` 等全局变量）。建议每次修改代码后同步更新 module.md。

**Q：跨模块修改时如何确定加载范围？**

A：1）读取项目级 CLAUDE.md 确定涉及模块；2）读取各模块 module.md 确定依赖；3）只加载涉及模块的代码 + 共享层接口摘要。

**Q：如果 module.md 声明的依赖与代码不一致怎么办？**

A：这是清单维护问题。建议定期用 `verification-loop` 技能验证清单与代码的一致性。不一致时以代码为准并更新清单。

### 🔧 故障排查

**Q：AI 仍然加载了整个项目怎么办？**

A：检查以下几点：1）CLAUDE.md 是否包含模块地图和加载策略；2）目标模块的 module.md 是否存在；3）AI 是否正确读取了 CLAUDE.md 的加载指令。

**Q：修改后其他模块报错怎么办？**

A：这可能是反向依赖遗漏。检查：1）修改模块的 module.md 反向依赖声明是否完整；2）是否检查了所有反向依赖模块的兼容性。

**Q：manifest.json 路径更新后扩展无法加载怎么办？**

A：检查：1）manifest.json 中所有路径是否指向实际存在的文件；2）content_scripts 的加载顺序是否保持原有依赖链（如 petManager.core.js 必须在子模块之前）。

## 技巧与提示

### 💡 实用技巧

- **渐进创建清单**：先为核心模块（pet、chat）创建 module.md，再逐步覆盖其他模块
- **清单验证**：每次修改代码后，用 grep 搜索验证 module.md 的依赖声明是否与代码一致
- **最小上下文原则**：AI 修改时，只要求加载 module.md 中声明的直接依赖，不要递归加载所有依赖的完整代码

### ⌨️ 快捷操作

| 操作 | 方法 | 说明 |
|------|------|------|
| 查看模块地图 | 读取 `CLAUDE.md` 模块地图章节 | 确定目标模块路径 |
| 查看模块依赖 | 读取 `modules/<name>/module.md` | 础认上下文范围 |
| 验证清单准确性 | `grep -r "window.PetManager" modules/<name>/` | 与 module.md 声明对比 |
| 检查反向依赖 | 读取 module.md 的反向依赖章节 | 确认影响范围 |

### 📚 最佳实践

- **清单先行**：修改代码前先读取 module.md，确认上下文范围后再开始
- **修改后更新**：代码修改完成后同步更新 module.md 的依赖声明
- **边界意识**：始终在模块边界内修改，跨模块修改必须显式声明影响范围
