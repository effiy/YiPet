## 功能介绍

本使用文档用于指导执行 `/generate-document init`，产出“项目级基础文档（10 个）+ `docs/项目初始化/01~07` 编号集”，并按动态检查清单做自检与回填。

**目标用户**：项目维护者、文档管理员、功能开发人员

## 快速开始

### 前置条件

- [x] 已进入 YiPet 仓库根目录（例如 `/Users/yi/Yi/YiPet/`）
- [x] 核心代码文件已存在（manifest.json、core/config.js 等）

### 30 秒上手

1. 执行 `/generate-document init`
2. 生成后按 `docs/项目初始化/05_动态检查清单.md` 完成 P0 自检并回填

## 操作场景

### 📋 场景1：首次项目初始化

**适用情况**：项目刚建立，还没有完整的文档体系，需要从零开始生成所有项目基础文件和全文档模板

**操作步骤**：
1. 执行 `/generate-document init`
2. 核对 `docs/` 是否存在 `network.md`、`state-management.md`（它们是 init 的基础文件清单的一部分）
3. 按 `05_动态检查清单.md` 执行 P0：存在性 + 互链一致性 + 证据化检查

**预期结果**：
- docs/ 目录创建成功
- 10 个项目基础文件完整生成
- docs/项目初始化/ 目录下 01-07 文档完整生成
- import-docs 同步完成（如有配置）
- wework-bot 通知发送成功（如有配置）

**注意事项**：
- 如果部分文件已存在：应更新事实类内容而非“存在即跳过”（详见 `.claude/skills/generate-document/rules/项目基础文件.md`）
- 无法证据化的内容必须标注 `> 待补充（原因：...）`，不要硬编

## 常见问题

### 💡 基础问题

**Q: 什么情况下需要执行项目初始化？**

**A**: 以下情况建议执行项目初始化：
- 新项目刚建立，还没有文档体系
- 项目结构发生重大变化，需要更新文档
- 需要为后续功能开发提供文档模板
- 加入新的团队成员，需要统一文档规范

**Q: 项目初始化会修改代码吗？**

**A**: 不会。generate-document 技能只负责生成文档，绝对不会修改任何代码内容。它只会创建或更新 docs/ 目录下的文档文件和根目录下的项目基础文件（CLAUDE.md、README.md）。

**Q: 执行项目初始化需要多长时间？**

**A**: 通常需要 1-3 分钟，取决于项目大小和网络情况。主要时间消耗在：
- 扫描项目代码结构
- 调用多个 agent 进行分析
- 生成多个文档文件
- 执行 import-docs 同步（如配置）
- 发送 wework-bot 通知（如配置）

---

### ⚙️ 高级问题

**Q: 如何自定义生成的文档内容？**

**A**: 主要通过以下方式：
- 修改 rules/ 下的规范文件，调整文档结构要求
- 修改 templates/ 下的模板文件（需求文档和需求任务可用模板）
- 在生成后手动编辑文档，补充"待补充"的内容
- 为项目添加更多事实源（如 package.json、.nvmrc 等）

**Q: 如何跳过 import-docs 或 wework-bot 步骤？**

**A**: 如果环境变量 API_X_TOKEN 未配置或 webhook 未设置，这些步骤会自动跳过并记录原因。你也可以手动执行这些步骤：
- import-docs：`node .claude/skills/import-docs/scripts/import-docs.js --dir docs --exts md`
- wework-bot：使用 `/wework-bot` 命令（详见对应技能文档）

**Q: 项目初始化的"6 步流程"具体是什么？**

**A**: 完整流程是：
1. 解析+规范检索：调用 spec-retriever 获取适用规范
2. 上游Grounding+影响分析：读取代码，调用 impact-analyst
3. 专家生成：调用 architect、planner、docs-lookup
4. 逐文档生成+自检：生成文档，调用 mermaid-expert、code-reviewer、quality-tracker
5. 保存+知识策展：保存文档，调用 knowledge-curator
6. 文档同步+通知：执行 import-docs，调用 wework-bot

### 🔧 故障排查

**Q: 执行 /generate-document init 报错怎么办？**

**A**: 请按以下步骤排查：
1. 检查是否在正确的项目目录下
2. 检查 .claude/ 目录是否完整，skills/ 和 agents/ 是否存在
3. 查看错误信息，确认是哪个环节失败
4. 如果是 agent 调用失败，检查对应的 agent 文件是否存在
5. 如果是文件写入失败，检查目录权限

**Q: 生成的文档中标注了很多"待补充"，怎么办？**

**A**: 这是正常的。"待补充"的原因通常包括：
- 未找到鉴权相关代码 → 需要手动补充认证方案
- 未找到安全相关代码 → 需要手动补充安全策略
- git 历史不完整 → 需要手动补充变更日志
- 未找到对应配置文件 → 需要手动补充相关内容

你可以根据实际情况，手动编辑文档补充这些内容。

**Q: Mermaid 图表渲染失败怎么办？**

**A**: 请检查：
- 图表语法是否正确（参考 mermaid 官方文档）
- 是否有未闭合的代码块
- 是否有特殊字符导致解析失败
- 可以使用 mermaid-expert agent 进行验证（已在生成流程中自动调用）

**Q: import-docs 同步失败怎么办？**

**A**: 请检查：
- API_X_TOKEN 环境变量是否正确配置
- 网络连接是否正常
- 远端文档服务是否可用
- 如果暂时无法同步，可以先跳过，后续手动执行

## 技巧与提示

### 💡 实用技巧

1. **提前准备事实源**：确保以下文件存在且内容完整，可以提高文档生成质量：
   - package.json（如有）
   - .nvmrc（如有）
   - .git 目录（用于 git log）

2. **善用参考文档**：
   - 生成新功能文档前，先看 docs/项目初始化/ 下的模板
   - 了解 rules/ 下的规范要求，可以预期文档结构
   - 参考已有的功能文档，保持风格一致

3. **分阶段验证**：
   - 先生成 01_需求文档.md，确认需求描述正确
   - 再生成 02_需求任务.md，确认任务分解合理
   - 最后生成完整文档集，避免重复修改

### ⌨️ 快捷键

项目初始化本身没有特殊快捷键，但你可以使用：
- `/generate-document init`：快速执行项目初始化
- `/generate-document 功能名-描述`：快速生成功能文档
- `/clear`：清空对话，重新开始

### 📚 最佳实践

1. **定期执行初始化**：建议在以下时机重新执行项目初始化：
   - 项目架构发生重大变化
   - 新增核心模块
   - 更新技术栈
   - 每次发布新版本后

2. **保持文档同步**：
   - 代码修改后，及时更新相关文档
   - 在 changelog.md 中记录变更
   - 定期运行 import-docs 同步文档

3. **团队协作约定**：
   - 统一使用 /generate-document 生成文档
   - 遵循 docs/architecture.md 中的架构约定
   - 遵循 CLAUDE.md 中的编码规范
   - 定期回顾和更新 FAQ.md
