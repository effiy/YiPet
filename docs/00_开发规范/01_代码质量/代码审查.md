# 代码审查规范

> 温柔陪伴助手 - 代码审查流程和规范

---

## 概述

本规范定义了温柔陪伴助手项目的代码审查流程和标准，确保代码质量、知识共享、项目一致性。

---

## 代码审查目的

### 1. 提升代码质量
- 发现潜在 bug 和问题
- 确保代码遵循最佳实践
- 提高代码可维护性

### 2. 知识共享
- 团队成员相互学习
- 了解项目各部分代码
- 统一技术方案

### 3. 项目一致性
- 统一编码风格
- 统一架构设计
- 统一技术栈使用

---

## 审查时机

### 必须审查
- 所有合并到 main 分支的代码
- 所有合并到 develop 分支的代码
- 重要功能变更
- 架构设计变更

### 建议审查
- 小功能修复
- 文档更新
- 重构代码

---

## 审查流程

### 1. 提交审查前

**作者自查**：
- [ ] 代码遵循编码规范
- [ ] 自测通过
- [ ] 已添加必要的测试
- [ ] 文档已更新
- [ ] 提交信息规范
- [ ] 没有遗留的调试代码

**准备说明**：
- 清晰的 PR 描述
- 关联相关 Issue
- 说明测试方法
- 提供必要的截图

### 2. 审查过程

#### 审查者选择
- 至少 1 名团队成员审查
- 相关模块负责人优先
- 重大变更需要多人审查

#### 审查关注点

**功能正确性**
- [ ] 代码逻辑正确
- [ ] 边界条件处理
- [ ] 错误处理完善
- [ ] 没有引入新 bug

**代码质量**
- [ ] 代码结构清晰
- [ ] 命名规范合理
- [ ] 注释充分必要
- [ ] 没有重复代码
- [ ] 函数/类大小适中

**架构设计**
- [ ] 符合项目架构
- [ ] 模块边界清晰
- [ ] 依赖关系合理
- [ ] 可扩展性考虑

**性能优化**
- [ ] 没有明显的性能问题
- [ ] 避免不必要的计算
- [ ] 资源合理使用

**安全性**
- [ ] 没有安全漏洞
- [ ] 输入验证充分
- [ ] 敏感数据处理正确

#### 审查反馈

**正面反馈**：
- 指出做得好的地方
- 给予鼓励和肯定
- 分享学习心得

**改进建议**：
- 使用建设性语气
- 具体指出问题
- 提供改进建议
- 解释为什么需要改进

**问题分类**：
- 🔴 必须修改：阻塞合并的严重问题
- 🟡 建议修改：建议改进的问题
- 🟢 可选修改：锦上添花的建议

### 3. 审查后处理

#### 作者回应
- 逐条回应审查意见
- 解释设计决策（如需要）
- 接受合理建议并修改
- 讨论不同意见（如需要）

#### 修改迭代
- 基于反馈修改代码
- 更新 PR 描述（如需要）
- 通知审查者再次审查
- 多次迭代直到达成一致

#### 合并代码
- 所有审查者批准
- CI 检查通过
- 解决所有讨论
- 选择合适的合并方式

## 审查标准

### 代码质量标准

#### 可读性
```javascript
// 好的代码
function calculateUserAge(birthDate) {
    const today = new Date();
    const birth = new Date(birthDate);
    let age = today.getFullYear() - birth.getFullYear();
    const monthDiff = today.getMonth() - birth.getMonth();

    if (monthDiff < 0 || (monthDiff === 0 && today.getDate() < birth.getDate())) {
        age--;
    }

    return age;
}

// 避免的代码
function calc(d) {
    const t = new Date();
    const b = new Date(d);
    let a = t.getFullYear() - b.getFullYear();
    const m = t.getMonth() - b.getMonth();
    if (m < 0 || (m === 0 && t.getDate() < b.getDate())) a--;
    return a;
}
```

#### 可维护性
```javascript
// 好的代码
class PetManager {
    constructor(options) {
        this.position = options.position;
        this.size = options.size;
    }

    updatePosition(newPosition) {
        this.validatePosition(newPosition);
        this.position = newPosition;
        this.notifyPositionChanged();
    }

    validatePosition(position) {
        if (!position.x || !position.y) {
            throw new Error('Invalid position');
        }
    }

    notifyPositionChanged() {
        this.emit('positionChanged', this.position);
    }
}

// 避免的代码
class PetManager {
    update(p) {
        if (!p) return;
        this.p = p;
        this.onChange && this.onChange(p);
    }
}
```

### 测试覆盖标准

- 新增功能必须有测试
- 修改功能需要更新测试
- 关键路径必须有集成测试
- 测试覆盖率要求：核心功能 ≥ 80%

## 审查示例

### 好的审查评论

```markdown
👍 这个实现很清晰！

有几个小建议：
1. 考虑将这个逻辑抽取成独立函数，提高复用性
2. 这个边界条件是否需要添加测试用例？
3. 变量命名可以更明确一些，比如 `userData` 改成 `userProfileData`

其他都做得很好，特别是错误处理部分！
```

### 需要改进的审查评论

```markdown
❌ 需要修改：

1. 这个逻辑有严重 bug，当输入为空时会崩溃
2. 没有遵循项目的命名规范
3. 缺少必要的错误处理
4. 没有添加对应的测试用例

请先修复这些问题，然后我再审查。
```

---

## 最佳实践

1. **及时审查**：收到审查请求后尽快响应
2. **小步审查**：PR 尽量小，便于审查
3. **尊重差异**：技术方案可能有多种，选择最合适的
4. **记录决策**：重要的技术决策要记录下来
5. **持续改进**：根据审查反馈不断优化
