## 功能介绍

本功能是继第一阶段重构后的第二阶段继续优化，将剩余的三个超大型模块（编辑器、Mermaid、AI 对话）按职责拆分为更小的文件，提升代码可维护性的同时保持 100% 功能兼容。作为用户，你不会感受到任何功能变化，但作为开发者，代码结构将更清晰。

🎯 **编辑器模块拆分**：核心逻辑与 UI 交互分离
⚡ **Mermaid 模块拆分**：渲染核心与 UI 交互分离
🔧 **AI 对话模块拆分**：API 封装与提示词管理分离

**谁应该阅读这份文档**：
- 项目维护者和开发者：了解新的模块组织方式
- 贡献者：知道在哪里添加新功能
- 任何对代码结构感兴趣的人

## 操作场景

### 📋 场景 1：在编辑器模块中添加新功能

**适用情况**：你需要为编辑器添加新的核心逻辑或 UI 功能

**操作步骤**：
1. 判断新功能属于"核心逻辑"还是"UI 交互"
   - 核心逻辑：数据处理、解析、计算等 → editor.core.js
   - UI 交互：DOM 操作、事件监听、弹窗等 → editor.ui.js
2. 在对应文件中添加新方法：
   ```javascript
   // 在 editor.core.js 中添加
   proto.myNewCoreMethod = function () {
     // 你的核心逻辑
   }
   
   // 或在 editor.ui.js 中添加
   proto.myNewUiMethod = function () {
     // 你的 UI 逻辑
   }
   ```
3. 测试新功能是否正常工作

**预期结果**：
- 新功能正常工作
- 代码按职责组织在正确的文件中
- 无 JS 错误

**注意事项**：
- ✅ 保持与现有代码风格一致
- ✅ 每个方法职责单一
- ❌ 不要在原兼容层文件中添加新代码

### 🎨 场景 2：在 Mermaid 模块中修改渲染逻辑

**适用情况**：你需要修改 Mermaid 图表的渲染行为

**操作步骤**：
1. 找到 mermaid.renderer.js 文件
2. 修改或添加渲染相关方法：
   ```javascript
   // 在 mermaid.renderer.js 中修改
   proto.processMermaidBlocks = async function (container) {
     // 你的渲染逻辑
   }
   ```
3. 如果需要修改 UI 交互，在 mermaid.ui.js 中修改
4. 测试 Mermaid 图表渲染是否正常

**预期结果**：
- 渲染逻辑更新成功
- 图表仍能正常显示
- 预览功能正常

**注意事项**：
- ✅ 修改前先备份原有逻辑
- ✅ 测试各种图表类型（流程图、时序图、甘特图等）
- ❌ 不要破坏向后兼容

### 🤖 场景 3：在 AI 模块中添加新的 API 功能

**适用情况**：你需要为 AI 对话添加新的 API 相关功能

**操作步骤**：
1. 判断新功能属于"API 封装"还是"提示词管理"
   - API 封装：请求、重试、流式处理等 → ai.api.js
   - 提示词管理：角色配置、系统提示词等 → ai.prompt.js
2. 在对应文件中添加代码
3. 测试 AI 对话功能是否正常

**预期结果**：
- 新功能正常工作
- AI 对话流程不受影响

**注意事项**：
- ✅ 保持 API 调用错误处理一致
- ✅ 测试流式响应
- ❌ 不要破坏现有配置存储

## 常见问题

### 💡 基础问题

**Q1：重构后功能有变化吗？**
A：没有变化。这是一次纯粹的代码组织重构，功能完全保持不变。

**Q2：我应该在哪个文件中添加新代码？**
A：
- 编辑器核心逻辑 → editor.core.js
- 编辑器 UI 交互 → editor.ui.js
- Mermaid 渲染 → mermaid.renderer.js
- Mermaid UI → mermaid.ui.js
- AI API 调用 → ai.api.js
- AI 提示词管理 → ai.prompt.js

**Q3：原文件还在吗？**
A：还在。原文件保留作为兼容层，确保向后兼容，不要在其中添加新代码。

**Q4：manifest.json 和 injectionService.js 为什么都要改？**
A：manifest.json 用于正常安装加载，injectionService.js 用于动态注入功能，两者需要保持一致。

### ⚙️ 高级问题

**Q5：如何判断一个方法属于 core 还是 ui？**
A：
- core：不直接操作 DOM，处理数据、计算、调用 API
- ui：直接操作 DOM、处理事件、显示弹窗等

**Q6：如果一个方法同时涉及 core 和 ui 怎么办？**
A：拆分为两个方法：
- core 方法处理逻辑
- ui 方法调用 core 方法并处理 UI
- 或者放在更核心的那一侧

**Q7：兼容层可以删除吗？**
A：暂时不可以。建议至少保留一个版本周期，确保所有用户都已升级。

### 🔧 故障排查

**Q8：控制台提示方法未定义怎么办？**
A：
1. 检查 manifest.json 加载顺序是否正确
2. 确认新文件在原文件之前加载
3. 检查 injectionService.js 是否同步更新
4. 重新加载扩展

**Q9：功能突然不工作了怎么办？**
A：
1. 检查控制台是否有 JS 错误
2. 确认方法是否正确挂载到原型
3. 回退到上一个版本验证是否是重构导致的
4. 检查浏览器缓存，尝试硬刷新（Ctrl+Shift+R）

**Q10：如何验证重构没有破坏功能？**
A：
1. 测试主要操作场景：编辑器、Mermaid、AI 对话
2. 检查控制台无错误
3. 对比重构前后的功能行为
4. 进行边界情况测试

## 技巧提示

### 💡 实用技巧

1. **使用文件查找快速定位**
   - 编辑器相关 → 搜索 "editor."
   - Mermaid 相关 → 搜索 "mermaid."
   - AI 相关 → 搜索 "ai."

2. **保持方法命名一致**
   - UI 相关方法以 ensure/show/close 等开头
   - 核心逻辑方法以操作名称开头

3. **在文件头部添加模块说明**
   ```javascript
   /**
    * 编辑器核心逻辑模块
    * 职责：数据处理、解析、工具函数
    */
   ```

### ⌨️ 快捷键（开发相关）

| 快捷键 | 功能 | 适用场景 |
|--------|------|---------|
| Ctrl+P (VSCode) | 快速打开文件 | 快速定位到 editor.core.js 等新文件 |
| Ctrl+Shift+F | 全项目搜索 | 查找某个方法在哪个文件中 |
| Ctrl+Shift+R | 硬刷新页面 | 重新加载扩展后刷新页面 |

### 📚 最佳实践

1. **先在新文件中开发，再逐步迁移**
   - 新功能直接在新文件中开发
   - 旧功能有修改时顺便迁移

2. **保持小步提交**
   - 每个模块拆分单独一个 PR
   - 每个 PR 有充分的测试说明

3. **文档同步更新**
   - 如有相关文档，同步更新文件路径说明
   - 为新文件添加必要的 JSDoc 注释
