## 功能介绍

本使用文档介绍了 YiPet 代码库重构的使用指南。重构工作主要解决超大型文件拆分、配置管理统一、错误处理统一等内容，目标是提升代码质量和可维护性。本次重构采用兼容优先策略，所有功能保持不变。

🎯 **向后兼容**：所有现有功能继续正常工作，用户感知不影响
⚡ **渐进式改善**：代码质量逐步提升，不影响现有开发进度
🔧 **低风险实施**：兼容层策略确保可以随时回滚

**谁应该使用这份文档**：
- YiPet 项目维护者
- 参与重构工作的开发者
- 想要了解代码结构的贡献者

## 操作场景

### 📋 场景 1：超大型文件拆分

**适用情况**：当你需要拆分 `petManager.session.js` 或其他超大型文件时使用。

**操作步骤**：

| 步骤 | 操作 | 说明 |
|------|------|------|
| 1 | 阅读代码 | 理解文件中的功能分组 |
| 2 | 设计拆分方案 | 按职责划分新文件结构 |
| 3 | 创建新文件夹 | `modules/pet/content/session/ |
| 4 | 迁移代码 | 将代码按组移动到新文件 |
| 5 | 更新 manifest.json | 在原文件前加载新文件 |
| 6 | 创建兼容层 | 原文件保留但内容简化 |
| 7 | 测试功能 | 验证所有功能正常 |

**预期结果**：
- ✅ 新文件结构清晰
- ✅ 所有功能正常工作
- ✅ 代码更易维护

**注意事项**：
- ✅ **正面提示：保持方法名和接口不变，确保向后兼容
- ❌ **负面警告：不要删除原文件，保留作为兼容层
- ✅ **正面提示：每个新文件使用 IIFE 模式，与现有代码风格一致
- ❌ **负面警告：不要改变现有功能的行为，除非有充分测试

### 🎨 场景 2：配置管理统一

**适用情况**：当你需要统一配置管理，合并 config.js 和 endpoints.js 时使用。

**操作步骤**：

| 步骤 | 操作 | 说明 |
|------|------|------|
| 1 | 分析现有配置 | 找出所有重复配置项 |
| 2 | 设计统一结构 | 在 config.js 中整合 |
| 3 | 增强 config.js | 添加端点配置和工具函数 |
| 4 | 创建兼容层 | endpoints.js 从 config.js 重新导出 |
| 5 | 验证配置读取 | 确保所有配置读取正常 |
| 6 | 测试 API 调用 | 确保所有 API 调用正常 |

**预期结果**：
- ✅ 配置集中管理
- ✅ 所有 API 调用正常
- ✅ 环境切换正常

**注意事项**：
- ✅ **正面提示**：保持 PET_CONFIG 全局变量不变
- ❌ **负面警告**：不要立即删除 endpoints.js，先保留兼容层
- ✅ **正面提示**：配置命名清晰的配置项的用途
- ❌ **负面警告**：不要修改配置的默认值，除非你确定

## 常见问题

### 💡 基础问题

**Q: 重构会影响现有功能吗？**
A: 不会。本次采用兼容层策略，所有现有功能继续正常工作。

**Q: 为什么需要兼容层？**
A: 兼容层确保重构可以平滑进行，降低风险，并允许随时回滚。

**Q: 原文件会被删除吗？**
A: 不会。原文件保留作为兼容层，确保现有代码继续工作。

**Q: 新代码的代码风格是什么？**
A: 保持与现有代码一致的风格：IIFE 模式，挂载到 window.PetManager.prototype。

### ⚙️ 高级问题

**Q: 如何判断一个文件是否需要拆分？**
A: 一般来说，超过 500 行的文件可以考虑拆分，超过 1000 行的文件建议拆分。

**Q: 拆分的原则是什么？**
A: 按单一职责原则拆分，每个文件负责一个明确的功能领域。

**Q: 兼容层需要保留多久？**
A: 建议至少保留一个版本周期，确认没有问题后再考虑移除。

**Q: 如何验证重构是否成功？**
A: 主要看两个方面：1) 所有功能继续正常工作；2) 代码结构更清晰，更易维护。

### 🔧 故障排查

**Q: 重构后某个功能不工作了怎么办？**
A: 首先检查：
1. 新文件是否在 manifest.json 中正确加载
2. 加载顺序是否正确（新文件在原文件之前）
3. 方法名和接口是否保持不变
4. 浏览器控制台是否有错误信息

**Q: 遇到合并冲突怎么办？**
A: 因为文件变小后，合并冲突会减少。如果遇到冲突，建议：
1. 先理解冲突的代码
2. 按功能分组解决冲突
3. 充分测试后再提交

**Q: 如何回滚重构？**
A: 如果需要回滚：
1. 从 git 恢复原文件
2. 从 manifest.json 移除新文件
3. 测试功能恢复正常

## 技巧提示

### 💡 实用技巧

1. **小步快跑**：每次只拆分一个文件，充分测试后再继续下一个
2. **保持兼容**：始终保持接口不变，这是安全重构的关键
3. **充分测试**：每次重构后都要充分测试，特别是修改的功能
4. **提交频繁**：每个小步骤都提交，便于回滚和审查
5. **文档同步**：同步更新相关文档，保持文档与代码一致

### ⌨️ 快捷键

本次重构不涉及快捷键变更。

### 📚 最佳实践

1. **先理解后重构**：在重构前充分理解代码功能
2. **保持简单**：不要过度工程化，保持代码简单清晰
3. **团队协作**：重构前与团队沟通，确保大家理解方案
4. **持续集成**：如果有 CI，确保重构通过所有测试
5. **代码审查**：重构后进行代码审查，确保质量
