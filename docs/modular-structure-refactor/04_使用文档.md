## 功能介绍

本指南面向 YiPet 项目开发者，说明重构后的目录结构如何使用和维护。重构后的目录结构与 YiWeb 代码结构规范对齐，分为 `cdn/`（共享资源+组件）、`modules/`（业务模块按 core/modules/features 三层）、`extension/`（扩展系统独立）三大区域。

- 🎯 **cdn/ 区域**：共享配置、工具、第三方库、样式、Vue 组件
- ⚡ **modules/ 区域**：按功能划分的业务模块，pet 模块有 core/modules/features 三层结构
- 🔧 **extension/ 区域**：Chrome 扩展系统（background、popup、消息路由）

## 快速开始

### 前置条件

- ✅ 已阅读 [设计文档](./03_设计文档.md) 了解重构方案
- ✅ 已完成目录重构变更（文件迁移和路径更新）

### 30秒上手

1. 打开 YiPet 项目根目录
2. 查看 `cdn/` 目录了解共享资源分布
3. 查看 `modules/` 目录了解业务模块组织
4. 查看 `extension/` 目录了解扩展系统结构
5. 完成 🎉 你已了解重构后的目录结构

## 操作场景

### 📋 场景1：定位配置文件

**适用情况**：需要修改 API 端点、宠物参数、环境配置时。

**操作步骤**：

| 步骤 | 操作 | 预期结果 |
|------|------|---------|
| 1 | 打开 `cdn/core/config.js` | 找到 `DEFAULT_CONFIG` 对象 |
| 2 | 修改 `api` 或 `pet` 中的配置值 | 配置生效 |
| 3 | 重新加载扩展（chrome://extensions → 重新加载） | 新配置生效 |

**注意事项**：
- ✅ `config.js` 是全局配置入口，所有模块通过 `PET_CONFIG` 读取配置
- ❌ 不要在其他文件中硬编码配置值

### 📋 场景2：添加新的功能模块到 pet 模块

**适用情况**：需要为宠物添加新功能（如"收藏"功能）时。

**操作步骤**：

| 步骤 | 操作 | 预期结果 |
|------|------|---------|
| 1 | 在 `modules/pet/modules/` 下创建 `petManager.bookmark.js` | 功能模块文件创建 |
| 2 | 在文件中定义功能并通过 `PetManager.prototype` 挂载 | 方法挂载到 PetManager |
| 3 | 在 `manifest.json` 的 `content_scripts.js` 中添加加载路径 | 扩展会加载新文件 |
| 4 | 在 `injectionService.js` 的 `CONTENT_SCRIPT_FILES` 中同步添加路径 | 动态注入列表同步更新 |
| 5 | 重新加载扩展 | 新功能可用 |

**注意事项**：
- ✅ 功能模块文件放在 `modules/pet/modules/` 目录下
- ❌ 不要放在 `modules/pet/core/`（那是核心实现）
- ❌ 不要放在 `modules/pet/features/`（那是特性文件）
- ✅ `manifest.json` 和 `injectionService.js` 的路径列表必须保持一致

### 🎨 场景3：添加新的 Vue 组件

**适用情况**：需要添加新的 UI 组件（如"收藏弹窗"）时。

**操作步骤**：

| 步骤 | 操作 | 预期结果 |
|------|------|---------|
| 1 | 在 `cdn/components/modal/` 下创建 `BookmarkModal/` 目录 | 组件目录创建 |
| 2 | 创建 `index.js`、`index.html` | 组件文件创建 |
| 3 | 在 `manifest.json` 的 `content_scripts.js` 中添加 `index.js` 路径 | 扩展会加载组件 |
| 4 | 在 `manifest.json` 的 `web_accessible_resources` 中添加 `index.html` 路径 | 模板可加载 |
| 5 | 重新加载扩展 | 新组件可用 |

**注意事项**：
- ✅ Vue 组件放在 `cdn/components/` 下，按类型分组
- ✅ 每个组件包含 `index.js`（逻辑）和 `index.html`（模板）
- ❌ 不要将组件放在 `modules/pet/` 下

### 📋 场景4：添加新的消息处理器

**适用情况**：需要在 background 中处理新的消息类型时。

**操作步骤**：

| 步骤 | 操作 | 预期结果 |
|------|------|---------|
| 1 | 在 `extension/background/actions/` 下创建处理器文件 | 处理器文件创建 |
| 2 | 在处理器中定义 `handleXxxAction` 函数 | 处理函数定义 |
| 3 | 在 `extension/background/bootstrap/imports.js` 中添加加载路径 | 扩展会加载处理器 |
| 4 | 在 `extension/messaging/messageRouter.js` 中注册处理器 | 消息路由配置 |
| 5 | 重新加载扩展 | 新消息类型可被处理 |

**注意事项**：
- ✅ 消息处理器放在 `extension/background/actions/` 下
- ✅ 消息路由放在 `extension/messaging/` 中

### 🎨 场景5：添加新的宠物角色

**适用情况**：需要添加新的宠物角色（如"程序员"）时。

**操作步骤**：

| 步骤 | 操作 | 预期结果 |
|------|------|---------|
| 1 | 在 `cdn/assets/images/programmer/` 下添加角色图标 | 角色图片资源添加 |
| 2 | 在 `modules/pet/modules/petManager.roles.js` 中添加角色配置（使用英文名） | 角色配置添加 |
| 3 | 在 `manifest.json` 的 `web_accessible_resources` 中添加图片路径 | 图片可加载 |
| 4 | 重新加载扩展 | 新角色可切换 |

**注意事项**：
- ✅ 角色图片目录使用英文名（如 `programmer/` 而非 `程序员/`）
- ✅ 角色配置在 `petManager.roles.js` 中统一管理

## 常见问题

### 💡 基础问题

**Q1: 为什么 `core/` 变成了 `cdn/core/`？**
A: 对齐 YiWeb 代码结构规范。在 YiWeb 项目中，共享的配置、常量、API、工具都放在 `cdn/` 目录下。

**Q2: 为什么 `core/utils/` 提升到了 `cdn/utils/`？**
A: YiWeb 规范中 `utils` 是 `cdn` 下的独立目录，与 `core` 同级。将 `utils` 从 `core` 下提升出来，使工具函数与核心配置平级，职责边界更清晰。

**Q3: 为什么 Vue 组件从 `modules/pet/components/` 移到了 `cdn/components/`？**
A: Vue 组件是跨模块共享的 UI 资源，不应隶属于某个特定业务模块。

**Q4: 为什么 `modules/extension/` 提升为顶级 `extension/`？**
A: 扩展系统代码是 Chrome Extension 的基础设施，与业务模块性质不同。

**Q5: 为什么角色图片目录改为英文名？**
A: 中文路径在某些系统、工具和终端中可能导致编码问题，使用英文名更稳定可解析。

### ⚙️ 高级问题

**Q6: 如何确保 manifest.json 和 injectionService.js 的路径列表一致？**
A: 每次更新 `manifest.json` 中 `content_scripts.js` 数组后，必须同步更新 `injectionService.js` 中 `CONTENT_SCRIPT_FILES` 静态属性。两者必须完全镜像，否则动态注入功能将失败。

**Q7: 动态加载的脚本路径如何更新？**
A: 动态加载的脚本通过 `chrome.runtime.getURL()` 获取路径。路径变更后需要更新以下文件：
- `modules/mermaid/page/load-mermaid.js` 中的 `mermaid.min.js` 路径
- `modules/session/page/load-jszip.js` 中的 `jszip.min.js` 路径

**Q8: 如何回退目录变更？**
A: 由于使用 `git mv` 执行文件移动，可通过 git 回退：
```bash
git log --oneline -5
git revert <commit-hash>
```

### 🔧 故障排查

**Q9: 扩展加载后出现"脚本加载失败"错误？**
A: 检查 `manifest.json` 中对应的脚本路径是否正确。常见原因：
- 路径中仍包含旧的 `core/`、`libs/`、`assets/` 前缀
- 文件实际位置与 manifest 中声明的不一致
- `manifest.json` 和 `injectionService.js` 路径不一致

**Q10: Vue 组件无法加载模板？**
A: 检查：
1. `manifest.json` 的 `web_accessible_resources` 是否包含组件模板路径
2. 组件 JS 中的模板加载路径是否正确
3. 模板文件是否实际存在于指定路径

**Q11: 角色图片无法显示？**
A: 检查：
1. `manifest.json` 的 `web_accessible_resources` 是否包含 `cdn/assets/images/*/icon.png` 路径
2. 角色配置中的角色名是否使用英文名（如 `teacher` 而非 `教师`）

## 技巧与提示

### 💡 实用技巧

1. **路径映射表**：执行目录变更前，先从 `manifest.json`、`imports.js`、`injectionService.js`、`web_accessible_resources` 和所有 `chrome.runtime.getURL` 调用中提取路径，生成映射表。

2. **分阶段验证**：按设计文档的四个阶段逐个验证，不要一次性迁移所有文件。

3. **git mv 而非 cp+rm**：使用 `git mv` 保留文件历史。

4. **双列表同步**：`manifest.json` 和 `injectionService.js` 的路径列表必须镜像一致。

5. **中文路径转英文**：角色图片目录从中文改为英文时，需同步更新 `config.js` 的默认角色名和所有引用中文角色名的代码。

### ⌨️ 快捷键

| 快捷键 | 功能 | 使用场景 |
|--------|------|---------|
| Ctrl+Shift+P (Mac: Cmd+Shift+P) | 切换宠物显示/隐藏 | 验证宠物功能 |
| Ctrl+Shift+X (Mac: Cmd+Shift+X) | 打开聊天窗口 | 验证聊天功能 |
| chrome://extensions → 重新加载 | 重新加载扩展 | 每次路径变更后 |

### 📚 最佳实践

1. **每次只改一个阶段**：先合并 cdn/，验证通过后再重组 pet/，避免大面积变更导致定位困难。

2. **变更后立即验证**：每次阶段变更后，立即重新加载扩展并检查所有功能是否正常。

3. **更新文档与代码同步**：变更完成后，同步更新 `CLAUDE.md` 和 `docs/structure.md`。
