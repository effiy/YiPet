## 功能介绍

项目初始化功能通过 `/generate-document init` 命令，为温柔陪伴助手 Chrome 扩展项目快速建立完整的文档体系，包括项目基础文件（CLAUDE.md、README.md、架构文档、变更日志等）和全文档编号集（需求、设计、使用、检查清单等）。所有文档基于规范驱动生成，确保结构一致性和内容可追溯性。

🎯 快速建立标准文档体系
⚡ 自动化文档生成流程
🔧 提供文档生成参考示例

**谁应该使用**：
- 新项目启动时需要建立文档体系的开发者
- 已有项目需要更新或标准化文档的维护者
- 需要生成功能文档但想先看示例的开发者

## 快速开始

### 前置条件

- [x] .claude/skills/generate-document/ 目录存在且包含完整规范
- [x] 项目有基本结构（manifest.json、核心代码目录等）
- [x] Claude Code 环境可用

### 30秒上手

1. **打开 Claude Code**，进入温柔陪伴助手项目根目录
2. **执行命令**：输入 `/generate-document init`
3. **等待完成**：工具会自动生成所有文档
4. **查看结果**：检查根目录和 docs/ 目录下的生成文件
5. **完成！** 🎉

## 操作场景

### 📋 场景：新项目首次初始化

**适用情况**：刚创建一个新的 Chrome 扩展项目，需要从零开始建立完整的文档体系。

**操作步骤**：

| 步骤 | 操作 | 说明 |
|------|------|------|
| 1 | 确认项目基本结构 | 确保有 manifest.json、core/ 或 src/ 目录、基本代码文件 |
| 2 | 打开 Claude Code | 进入项目根目录 |
| 3 | 执行 `/generate-document init` | 等待工具完成，通常需要 1-2 分钟 |
| 4 | 查看生成的文件 | 检查根目录的 CLAUDE.md、README.md，docs/ 目录下的文档 |
| 5 | 根据实际情况调整 | 修改文档中"待补充"的部分，补充项目特有内容 |

**预期结果**：
- 根目录下有完整的 CLAUDE.md 和 README.md
- docs/ 目录下有 architecture.md、changelog.md、devops.md、FAQ.md、auth.md、security.md
- docs/项目初始化/ 目录下有 01-07 完整文档链
- 所有文档结构规范，内容基于项目实际情况

**注意事项**：
✅ 首次生成建议完整阅读所有文档，确认内容符合项目情况
❌ 不要直接删除生成的文档，应基于其进行修改

### 🎨 场景：更新已有项目文档

**适用情况**：项目已有一段时间，需要更新文档以反映最新架构、新增功能或变更记录。

**操作步骤**：

| 步骤 | 操作 | 说明 |
|------|------|------|
| 1 | 备份现有文档 | 如需要保留自定义内容，先备份 docs/ 目录 |
| 2 | 执行 `/generate-document init` | 工具会覆盖已有基础文档 |
| 3 | 对比新旧差异 | 使用 git diff 查看变化，选择性保留自定义内容 |
| 4 | 合并自定义内容 | 将备份中的有用内容合并到新文档中 |
| 5 | 更新技术事实 | 检查文档中引用的架构、代码路径是否与最新状态一致 |

**预期结果**：
- 文档结构保持最新规范
- 项目特有内容被保留
- 变更日志包含最新提交
- 架构文档反映当前项目结构

**注意事项**：
✅ 执行前建议先提交当前状态，便于对比和回退
❌ 不要覆盖重要的自定义文档内容

### 🔧 场景：为新功能生成文档（参考示例）

**适用情况**：需要为新功能（如"宠物动画优化"）生成完整文档，想先参考项目初始化的文档结构。

**操作步骤**：

| 步骤 | 操作 | 说明 |
|------|------|------|
| 1 | 查看 docs/项目初始化/ 示例 | 阅读 01-07 文档，了解完整结构 |
| 2 | 整理新功能需求 | 明确功能的用户故事、验收标准、使用场景 |
| 3 | 执行 `/generate-document 功能名-描述` | 如 `/generate-document 宠物动画优化-提升动画流畅度` |
| 4 | 按规范填写内容 | 基于需求和代码实现，完善各文档内容 |
| 5 | 执行自检 | 使用动态检查清单验证文档完整性 |

**预期结果**：
- docs/宠物动画优化/ 目录下有完整的 01-07 文档
- 文档基于实际代码实现，无虚构内容
- 可用于后续开发参考和交接

**注意事项**：
✅ 功能名应简洁明了，描述应说明功能价值
❌ 不要在文档中添加无来源的技术事实

## 常见问题

### 💡 基础问题

**Q: `/generate-document init` 和普通功能文档生成有什么区别？**

A: `init` 命令专门用于项目初始化，会生成 8 个项目基础文件 + docs/项目初始化/ 下的 01-07 全文档（包括 init 特有的 06_实施总结）。普通功能文档生成只创建 docs/功能名/ 下的 01-05 和 07，不包含 06。

**Q: 执行 init 会覆盖我已有的文档吗？**

A: 是的，会覆盖根目录的 CLAUDE.md、README.md 以及 docs/ 下的基础文档。建议执行前先提交当前状态或备份重要内容。

**Q: 生成的文档中有很多"待补充"，这是正常的吗？**

A: 这是正常的。防幻觉原则要求无来源的内容必须标注"待补充"而不是虚构。你可以根据项目实际情况补充这些内容。

### ⚙️ 高级问题

**Q: 我想自定义文档生成规则，应该怎么做？**

A: 修改 .claude/skills/generate-document/rules/ 目录下的对应规范文件。规范是唯一契约，模板仅作辅助（设计文档和动态检查清单禁用模板）。

**Q: 如何添加新的 Agent 参与文档生成？**

A: 在 .claude/agents/ 目录下添加新的 Agent 描述文件，然后在 generate-document/SKILL.md 的对应环节调用它。

**Q: 项目报告中的"变更前后对比"是如何生成的？**

A: 基于 git diff 或文件系统的实际变更生成。如果是新文件，变更前标注"（新增文件，无变更前内容）"。

### 🔧 故障排查

**Q: 执行 `/generate-document init` 时报错"找不到规范文件"，怎么办？**

A: 检查 .claude/skills/generate-document/ 目录是否存在，以及 rules/ 目录下是否有必需的规范文件。如果缺失，可能需要更新 .claude 子模块。

**Q: import-docs 同步失败，会影响主流程吗？**

A: 不会。根据规范，导入失败不阻断流程，会记录失败状态并继续执行 wework-bot 通知。你可以后续手动同步或排查原因。

**Q: wework-bot 通知发送失败，怎么办？**

A: 检查 webhook 配置是否正确。通知失败也会记录到项目报告中，不影响文档生成的主流程。

## 技巧提示

### 💡 实用技巧

1. **先预览再执行**：在执行 `/generate-document init` 前，先阅读 .claude/skills/generate-document/rules/ 下的规范，了解会生成什么内容。

2. **善用 git 对比**：执行前后使用 `git diff` 查看变化，便于选择性保留自定义内容。

3. **分批补充"待补充"**：生成的文档中会有很多"待补充"，不必一次性全部填完，可以在后续开发中逐步完善。

4. **将示例文档作为模板**：docs/项目初始化/ 下的文档可以作为后续功能文档的参考模板。

5. **定期更新文档**：项目有重大架构变更时，建议重新执行 `/generate-document init` 保持文档同步。

### ⌨️ 快捷键（如适用）

本文档功能不涉及快捷键。

### 📚 最佳实践

1. **文档与代码同步变更**：修改代码时，同步考虑是否需要更新对应文档。

2. **保持防幻觉习惯**：写文档时，不确定的内容标注"待补充"而不是猜测。

3. **定期执行自检**：使用动态检查清单验证文档完整性。

4. **知识策展**：将项目特有经验更新到 .claude/agents/memory/ 中，便于后续 AI 参考。
