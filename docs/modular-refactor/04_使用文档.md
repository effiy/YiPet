## 功能介绍

本使用指南面向 YiPet 项目开发者，说明如何执行目录结构模块化重构。重构将项目结构按 YiWeb 规范对齐，使目录命名一致、分层清晰、职责明确，同时保证 Chrome Extension 功能完整性。

- 🎯 **4 阶段渐进式重构**：cdn/ 合合 → pet 重组 → extension 提升 → 英文化，每步可独立验证
- ⚡ **路径集中管理**：所有路径变更通过 manifest.json、imports.js、injectionService.js 三处管理
- 🔧 **可回退**：每阶段变更可独立回退，不影响其他阶段

## 快速开始

### 前置条件

- ✅ YiPet 项目代码完整（core/、libs/、assets/、modules/ 四大目录存在）
- ✅ Chrome 浏览器可加载扩展（开发者模式）
- ✅ 理解 manifest.json 中 content_scripts.js 加载顺序的含义
- ✅ 了解 PetManager 功能链：PET_CONFIG → token → request → ApiManager → Services → core → modules → features → entry

### 30 秒上手

1. 读取 `manifest.json`、`imports.js`、`injectionService.js` 中所有路径
2. 生成旧→新路径映射表
3. 按阶段1（cdn/ 合并）开始执行目录迁移
4. 更新三处路径集中入口
5. 在 Chrome 中加载扩展验证功能 🎉

## 操作场景

### 📋 场景 1：合并核心资源到 cdn/

**适用情况**：需要将分散的 core/、libs/、assets/ 目录合并到统一的 cdn/ 目录下。

**操作步骤**：

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | 创建 `cdn/core/`、`cdn/utils/`、`cdn/libs/`、`cdn/assets/`、`cdn/components/` 目录 | 目录结构就绪 |
| 2 | 将 `core/config.js`、`core/bootstrap/`、`core/constants/`、`core/api/` 移入 `cdn/core/` | `cdn/core/` 包含配置+引导+常量+API |
| 3 | 将 `core/utils/` 整体移入 `cdn/utils/` | `cdn/utils/` 包含所有工具函数 |
| 4 | 将 `libs/` 整体移入 `cdn/libs/` | `cdn/libs/` 包含 7 个第三方库 |
| 5 | 将 `assets/` 整体移入 `cdn/assets/` | `cdn/assets/` 包含样式+图片+图标 |
| 6 | 将 `modules/pet/components/` 移入 `cdn/components/` | `cdn/components/` 包含 Vue 组件 |
| 7 | 更新 `manifest.json` 中所有路径（`core/` → `cdn/core/`/`cdn/utils/`、`libs/` → `cdn/libs/`、`assets/` → `cdn/assets/`、`modules/pet/components/` → `cdn/components/`） | 路径全部更新 |
| 8 | 更新 `imports.js` 中 `core/` → `cdn/core/` 路径 | importScripts 路径更新 |
| 9 | 更新 `injectionService.js` 中 CONTENT_SCRIPT_FILES 路径 | 动态注入路径更新 |
| 10 | 在 Chrome 中加载扩展验证 | ✅ 扩展加载无 404 错误 |

**预期结果**：`core/`、`libs/`、`assets/` 目录不再存在，所有内容在 `cdn/` 下按职责分层。

**注意事项**：
- ✅ 保持 manifest.json 中脚本加载顺序不变
- ❌ 不要修改脚本内部逻辑，只改路径

## 常见问题

### 💡 基础问题

**Q: 为什么要把 core/、libs/、assets/ 合并到 cdn/？**

A: YiWeb 项目规范使用 `cdn/` 作为共享资源根目录。当前 `core/`、`libs/`、`assets/` 职责分散且命名不一致（`core/utils/api/` 与 `core/api/` 职责重叠）。合并后与 YiWeb 体系一致，减少顶级目录数量。

**Q: 为什么去掉 `modules/pet/content/` 中间层？**

A: `content/` 是 Chrome Extension 的概念（content script），不应出现在目录结构中作为中间层。去掉后按 core/modules/features 三层组织更清晰。

**Q: Vue 组件为什么要提升到 `cdn/components/`？**

A: Vue 组件是共享 UI 资源，不应嵌套在 `modules/pet/components/` 业务模块内部。提升后多个模块可复用组件。

### ⚙️ 高级问题

**Q: 如何保证脚本加载顺序不被破坏？**

A: 重构只改变文件物理路径，不改变 manifest.json content_scripts.js 中脚本的加载顺序。PetManager 功能链的关键依赖顺序（PET_CONFIG → token → request → … → core → modules → features → entry → bootstrap/index）保持不变。

**Q: imports.js 中的路径和 manifest.json 有什么区别？**

A: imports.js 是 background service worker 的脚本加载入口（使用 `importScripts`），manifest.json content_scripts.js 是 content script 的加载入口。两者路径需同步更新。

**Q: 如何处理 `chrome.runtime.getURL()` 动态加载路径？**

A: 所有使用 `chrome.runtime.getURL()` 动态加载的路径（如 mermaid 脚本、jszip 脚本、Vue 模板 HTML）都需要按路径映射表更新。搜索所有 getURL 调用确认。

### 🔧 故障排查

**Q: 扩展加载后宠物不显示怎么办？**

A: 检查 manifest.json content_scripts.js 中 petManager.core.js 的路径是否正确（应为 `modules/pet/core/petManager.core.js`）。如果 PetManager 未定义，说明加载顺序或路径有问题。

**Q: 组件 HTML 模板加载 404 怎么办？**

A: 检查 manifest.json web_accessible_resources 中模板路径是否更新为 `cdn/components/` 前缀。ChatWindow 的 HTML 模板路径应为 `cdn/components/chat/ChatWindow/index.html`。

**Q: 角色图片加载失败怎么办？**

A: 检查 `cdn/assets/images/` 下目录名是否已改为英文名（doctor/teacher/chef/police），检查 `petManager.roles.js` 中角色配置的图片路径引用是否已更新。

## 技巧与提示

### 💡 实用技巧

1. **路径映射表先行**：执行目录变更前，先从 manifest.json、imports.js、injectionService.js 和所有 `chrome.runtime.getURL` 调用中提取路径，生成映射表。
2. **分阶段验证**：每完成一个阶段就在 Chrome 中验证一次，不要等全部完成再验证。
3. **保留旧目录备份**：在删除旧目录前先确认新目录功能正常。
4. **全局变量不变**：PET_CONFIG、PetManager、StorageHelper 等全局变量名不变，只改物理路径。

### 📚 最佳实践

1. **路径变更最小化**：优先使用批量替换前缀方式（`core/` → `cdn/core/`），而非逐行修改。
2. **加载顺序不动**：manifest.json content_scripts.js 的脚本顺序是功能链的关键，只改路径不改顺序。
3. **验证完整**：每个阶段完成后验证全部功能，不只是当前阶段涉及的功能。
