# 模块设计规范

> 温柔陪伴助手 - 模块设计和架构原则

---

## 概述

本规范定义了温柔陪伴助手项目的模块设计原则和架构设计方法，确保模块设计合理、职责清晰、易于维护和扩展。

---

## 模块设计原则

### 1. 单一职责原则 (SRP)
- 每个模块只负责一个功能领域
- 避免模块功能过于复杂
- 模块内部高内聚，模块之间低耦合

### 2. 开闭原则 (OCP)
- 对扩展开放，对修改关闭
- 通过添加新模块来扩展功能
- 避免修改现有模块的核心代码

### 3. 接口隔离原则 (ISP)
- 模块接口设计要精简
- 避免依赖不需要的接口
- 提供明确的模块边界

### 4. 依赖倒置原则 (DIP)
- 依赖抽象而非具体实现
- 使用接口定义模块契约
- 降低模块之间的耦合度

## 模块分类

### 核心模块
- **职责**：提供基础服务和基础设施
- **特点**：稳定性高，变更频率低
- **示例**：配置管理、API 集成、工具函数

### 功能模块
- **职责**：实现具体的业务功能
- **特点**：根据需求变更，变更频率中等
- **示例**：宠物管理、聊天功能、截图功能

### 扩展模块
- **职责**：提供可选的增强功能
- **特点**：可独立启用/禁用，变更频率高
- **示例**：Mermaid 渲染、特殊角色

## 模块间通信

### 1. 事件总线模式
```javascript
// 事件总线
class EventBus {
    constructor() {
        this.events = {};
    }

    on(event, callback) {
        if (!this.events[event]) {
            this.events[event] = [];
        }
        this.events[event].push(callback);
    }

    emit(event, data) {
        if (this.events[event]) {
            this.events[event].forEach(callback => callback(data));
        }
    }
}
```

### 2. 直接调用
- 核心模块可被功能模块直接调用
- 功能模块之间通过事件通信
- 避免功能模块之间的直接依赖

### 3. 状态共享
- 核心模块管理共享状态
- 功能模块通过核心模块访问状态
- 状态变更通过事件通知

## 模块生命周期

### 1. 初始化阶段
- 模块构造函数执行
- 依赖注入完成
- 初始状态设置

### 2. 启动阶段
- `init()` 方法调用
- 事件监听器注册
- 资源加载

### 3. 运行阶段
- 处理业务逻辑
- 响应事件
- 状态管理

### 4. 销毁阶段
- 清理资源
- 移除事件监听器
- 保存状态

## 最佳实践

1. **先设计后实现**：在编码前先设计模块结构
2. **保持模块精简**：避免模块过于庞大
3. **明确模块边界**：定义清晰的模块接口
4. **文档化模块设计**：记录模块设计决策
5. **及时重构**：发现设计问题及时重构
