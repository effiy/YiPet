## 功能概述

重构 YiPet 项目目录结构，采用模块化开发架构，使每个模块形成独立的内容边界。核心目标是降低 AI coding 的上下文膨胀问题——当 AI 修改某个模块时，只需加载该模块及其直接依赖的文件，而非整个项目的 ~19000 行 PetManager 代码和 80+ 个 content_scripts 入口。通过目录边界、模块清单（module.md）和 CLAUDE.md 分层约束机制，确保每次任务的范围可控、依赖可追溯、影响可验证。

🎯 **上下文收敛**：模块化后单次任务上下文从全项目缩减至 1-3 个模块
⚡ **边界隔离**：每个模块拥有独立目录、清单、入口，改动不扩散
📖 **依赖透明**：模块间依赖通过清单文件显式声明，AI 可精确判断影响范围

## 验收标准

### P0 - 必须通过

1. **module.md 依赖清单**：每个模块目录包含 `module.md`，列出模块名、职责、入口文件、上游依赖、反向依赖、导出接口
2. **上下文收敛**：单模块任务的 AI 上下文不超过 3000 行代码（当前 petManager 约 19000 行）
3. **CLAUDE.md 模块地图**：项目根 CLAUDE.md 包含模块索引表和上下文加载策略
4. **manifest.json 兼容**：重构后 manifest.json 的 content_scripts 和 background/service_worker 路径仍可正常工作

### P1 - 应该通过

1. **模块级 CLAUDE.md**：核心模块（pet、chat、session）拥有独立的 CLAUDE.md
2. **影响链可追溯**：从 module.md 出发可在 2 步内追踪到跨模块影响
3. **目录命名一致**：模块目录命名与 manifest.json 中的路径命名一致

### P2 - 可以有

1. **manifest.json 自动推导**：content_scripts 列表可从模块目录结构和 module.md 自动生成
2. **模块热插拔**：模块可通过配置启用/禁用，不影响其他模块运行

## 功能详情

### 模块化目录结构

**价值**：将 ~19000 行的 PetManager 单体拆分为独立模块，每个模块有明确边界

**解决的痛点**：当前 PetManager 的 22 个子模块文件（petManager.*.js）通过 `window.PetManager.prototype` 混合挂载，所有文件必须同时加载才能运行，AI 无法按需读取

### module.md 依赖清单机制

**价值**：显式声明模块间依赖，替代隐式的 window 全局变量引用链

**解决的痛点**：当前依赖关系散布在各文件的 `window.PetManager` 引用中，无法从目录结构推断

### CLAUDE.md 层级约束

**价值**：不同粒度的任务自动匹配不同深度的上下文，避免过度加载

**解决的痛点**：AI 每次任务都倾向于读取整个项目，导致上下文膨胀和成本浪费

## 使用场景示例

📋 **场景 1：单模块修改**
- **背景**：开发者要求 AI 修改聊天窗口的 UI 组件
- **操作**：AI 读取模块地图 → 确定 chat 模块 → 读取 `modules/chat/module.md` → 只加载 chat 模块文件
- **结果**：上下文仅包含 chat 模块（~2000 行），而非全项目

🎨 **场景 2：跨模块重构**
- **背景**：开发者要求重构 API 调用层，影响 chat 和 session
- **操作**：AI 读取项目级 CLAUDE.md → 找到 api 模块 → 通过 module.md 找到依赖模块 pet、chat、session → 加载这 4 个模块
- **结果**：影响范围精确可控，无遗漏无冗余

🔍 **场景 3：新增模块**
- **背景**：开发者要添加"通知"功能模块
- **操作**：按 module.md 模板创建目录 → 在 CLAUDE.md 模块地图中注册 → 在 core 共享层声明新依赖
- **结果**：新模块自动获得上下文边界约束，AI 可按需加载
