## 主要操作场景

### 场景 1：新开发者第一次接入仓库

- **前置条件**：仓库可读；扩展可在 Chrome 中加载。
- **操作步骤**：
  - 读 `README.md`（如何加载扩展/快速开始）
  - 读 `docs/architecture.md`（目录/模块/加载顺序）
  - 读 `docs/network.md` + `docs/state-management.md`（网络与状态约定）
- **预期结果**：能在不读大量源码的前提下，理解入口、核心模块、网络与持久化边界。

### 场景 2：生成新功能文档（简洁功能名 + 一句话描述）

- **操作步骤**：
  - 执行 `/generate-document <功能名>-<一句话描述>`
  - 参考 `docs/项目初始化/` 的编号集结构与写法（特别是证据化写法与“待补充”标注）
- **预期结果**：新功能文档集结构统一、可追溯、可复核。

### 场景 3：项目结构变化后 re-init 更新

- **操作步骤**：再次执行 `/generate-document init`，刷新“事实类章节”（目录树、入口、关键路径、网络/状态约定等）。
- **预期结果**：基础文档与真实代码保持同步；冲突点以 `> 待确认（原因：...）` 标注并给出证据路径。

## 验收标准

### P0（必须）

- 10 个项目基础文件在仓库中存在且非空（见“范围与产物”）
- `docs/项目初始化/01~07` 存在且非空
- 文档中引用的**代码路径**在仓库内真实存在；无法证据化的内容必须写 `> 待补充（原因：...）`

### P1（应该）

- `docs/network.md` 与 `docs/auth.md` 对“Token 来源/存储/传递方式（Header 名称）”描述一致
- `docs/state-management.md` 对持久化的描述与 `core/bootstrap/bootstrap.js` 的 `StorageHelper` 能互相印证
